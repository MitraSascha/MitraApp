## Was wurde getan

Live-Testing der App hat mehrere kritische Bugs aufgedeckt. Alle behoben am 2026-04-28.

## Bug 1: HERO Sync 400 Bad Request

**Ursache:** GraphQL Query verwendete falsche Feldnamen (geraten statt per Introspection ermittelt).

**Fix:** Schema-Introspection gegen echten HERO Endpoint durchgeführt.
Korrekturen in `backend/apps/termine/hero_sync.py`:
- `start_at` / `end_at` → `start` / `end`
- `partnerId: ID` → `partner_ids: [Int]`
- `from/to: String` → `start/end: DateTime` (ISO mit `+HH:MM` Timezone)
- `status` / `type` → `is_done` (Bool) / `category.name` (String)
- `address` / `customer` entfernt (nicht in CalendarEvent vorhanden)

## Bug 2: HERO Zeiten 2 Stunden falsch

**Ursache:** HERO gibt Berliner Lokalzeit mit `+00:00` zurück (API-Bug auf HERO-Seite).
Django speicherte es als UTC → Browser zeigte +2h zu viel.

**Fix:** `_fix_hero_datetime()` in `hero_sync.py`:
```python
naive = datetime.fromisoformat(dt_str.replace('+00:00', ''))
return naive.replace(tzinfo=ZoneInfo('Europe/Berlin')).isoformat()
```
Angewandt auf `beginn` und `ende` beim `update_or_create`.

**Wichtig:** Bereits falsch gespeicherte Termine müssen neu gesynct werden (update_or_create überschreibt via hero_crm_id).

## Bug 3: Frontend Termine auf falschem Tag

**Ursache:** `toISOString().slice(0, 10)` gibt UTC-Datum zurück. In CEST (UTC+2) ist
Mitternacht lokal = 22:00 UTC Vortag → Tag-Keys waren um 1 Tag verschoben.
Außerdem `t.beginn.startsWith(key)` schlägt fehl wenn `beginn` im UTC-Format (`Z`) ist.

**Fix in `mitra-app/src/app/features/termine/stores/termine.store.ts`:**
```typescript
function toLocalDateKey(d: Date): string {
  return `${d.getFullYear()}-${String(d.getMonth()+1).padStart(2,'0')}-${String(d.getDate()).padStart(2,'0')}`;
}
// Verwendung:
const key = toLocalDateKey(tag);
.filter(t => toLocalDateKey(new Date(t.beginn)) === key)
```

**Fix in `termine-liste.component.ts`:**
```typescript
readonly heutigesDatum = (() => {
  const d = new Date();
  return `${d.getFullYear()}-${String(d.getMonth()+1).padStart(2,'0')}-${String(d.getDate()).padStart(2,'0')}`;
})();
```

## Bug 4: lese-visitenkarte 500 Internal Server Error

**Ursachen (mehrere):**
1. `claude` CLI nicht im Docker-Container installiert
2. `~/.claude` als read-only gemountet → Claude CLI kann keine Session-Dateien schreiben (ENOENT)
3. `lese_visitenkarte()` bettete Base64 als Text in Prompt ein — Vision funktioniert so nicht über CLI

**Fix:**
- Node.js 20 + Claude CLI manuell im laufenden Container installiert (im Dockerfile bereits vorbereitet, Build scheitert aktuell an apt-get Netzwerkfehler)
- `_ensure_claude_home()` in `claude_service.py`: kopiert `/root/.claude` einmalig nach `/tmp/claude_home/` (beschreibbar), Claude CLI mit `HOME=/tmp/claude_home`
- `lese_visitenkarte()` nutzt `@<dateipfad>` Syntax im Prompt für Bild-Referenz

**Testbestätigt:** `claude -p 'Sag hallo'` antwortet korrekt aus Container heraus.
